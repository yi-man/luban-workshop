## 1. 项目背景与目标

### 1.1 背景
GLM Coding Plan 采用限量发售机制，每日 10:00 释放库存。由于用户量激增，库存通常在 1-3 秒内被抢光。

### 1.2 目标
- **核心目标**: 在库存释放后 3 秒内完成从检测→点击→验证→锁单的全流程
- **速度指标**:
  - 库存检测: <50ms/次
  - 从检测到点击: <500ms
  - 滑块验证: <1s (AI自动)
  - 总耗时: <3s

### 2.2 组件职责

#### StockMonitor (库存监控器)
- **核心职责**: 以最低延迟检测库存释放
- **关键实现**:
  - 使用 `aiohttp` + `TCPConnector` 预建立连接
  - 请求超时设置为 `500ms`（快速失败）
  - 轮询间隔 `20ms`（每秒50次）
  - 检测到库存后立即通过 `asyncio.Event` 通知浏览器组件

#### BrowserController (浏览器控制器)
- **核心职责**: 执行购买操作（点击、验证）
- **关键实现**:
  - 使用 `Playwright` 控制 Chromium
  - 浏览器实例提前启动并登录
  - 页面预滚动到购买区域
  - 收到库存信号后 `0.5秒内` 完成点击

#### CaptchaSolver (验证码求解器)
- **核心职责**: AI自动识别滑块缺口并模拟拖动
- **关键实现**:
  - 使用 `OpenCV` 进行图像处理（边缘检测）
  - 使用轻量级CNN模型定位缺口
  - 生成符合人类行为的贝塞尔曲线轨迹
  - **失败兜底**: 3次失败后，打印提示并播放提示音，等待人工介入

## 4. 核心设计决策

### 4.1 速度优化决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **库存检测方式** | API轮询（非浏览器） | 50ms/次 vs 2s/次，快40倍 |
| **轮询频率** | 50次/秒（20ms间隔） | 平衡速度与防封，每秒50次理论上可检测 |
| **浏览器策略** | 预热常驻（非即用即启） | 省去2-3秒启动时间，点击延迟<500ms |
| **连接管理** | TCP连接池预热 | 避免HTTP握手的200ms延迟 |
| **滑块验证** | 本地AI（非打码平台） | 省去1-3秒网络延迟，本地<500ms |

### 4.2 可靠性决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **滑块失败兜底** | 3次AI尝试后人工介入 | 平衡自动化与成功率，避免无限重试浪费时间 |
| **时间同步** | NTP同步到毫秒级 | 确保10:00:00精准触发，避免本地时钟漂移 |
| **超时策略** | 快速失败（500ms超时） | 不等待无响应请求，立即重试 |
| **登录检测** | 独立命令`check-login` | 抢购前可验证登录态，避免抢购时才发现未登录 |

### 4.3 资源与成本决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **部署方式** | 本地运行（非云端） | 用户要求暂不上云，避免额外成本 |
| **账号策略** | 单账号（非多开） | 用户明确不需要多账号并发 |
| **AI模型** | 轻量级本地模型（非云端API） | 零成本，无网络延迟 |
| **打码平台** | 不使用（除非AI完全失败） | 避免账号和充值成本 |

## 5. 接口设计

### 5.1 CLI 命令

```bash
# 1. 检测登录状态（抢购前运行）
python -m glm_coding_bot check-login
# 输出: ✅ 登录状态有效 (用户名: xxx) 或 ❌ 未登录或登录已过期

# 2. 执行抢购
python -m glm_coding_bot buy \
  --product Max \           # 可选: Lite, Pro, Max (默认Max)
  --time 10:00:00 \         # 抢购时间 (默认10:00:00)
  --poll-interval 0.02      # 轮询间隔秒 (默认0.02=20ms)

# 3. 快速测试（验证各组件是否工作）
python -m glm_coding_bot quick-test
```

### 5.2 Python API

```python
from glm_coding_bot import StockMonitor, BrowserController, CaptchaSolver

# 高级用法：自定义流程
async def custom_flow():
    # 1. 启动库存监控
    monitor = StockMonitor(
        product_id='product-047',
        poll_interval=0.02,  # 20ms
        cookies_file='cookies.json'
    )
    
    # 2. 预热浏览器
    browser = BrowserController()
    await browser.preheat()
    
    # 3. 等待库存信号
    has_stock = await monitor.wait_for_stock(timeout=120)
    
    if has_stock:
        # 4. 执行购买
        success = await browser.click_buy('Max')
        
        if success:
            # 5. 处理验证码
            captcha = CaptchaSolver()
            result = await captcha.solve_with_fallback(
                max_ai_attempts=3,
                on_manual=lambda: print("请手动完成滑块验证...")
            )
```

|------|--------|----------------|------|

## 7. 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| **滑块AI识别失败** | 中 | 高 | 3次失败后转人工，播放提示音提醒 |
| **IP被封** | 低 | 高 | 控制轮询频率<50次/秒，添加随机抖动 |
| **登录态过期** | 中 | 高 | 提供`check-login`命令提前检测 |
| **时间不同步** | 中 | 高 | 抢购前NTP同步，误差<10ms |
| **浏览器预热失败** | 低 | 中 | 启动时检测，失败则报错退出 |
| **AI模型加载失败** | 低 | 中 | 首次运行时自动下载模型文件 |
